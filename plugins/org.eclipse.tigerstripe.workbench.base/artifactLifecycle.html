Artifacts Lifecycle

As part of a the API-refactoring, the lifecycle of Artifacts has been revisited. The existing API had evolved into a very complicated series of interactions between the Artifact Mgr and its clients.

The lifecycle of Artifacts now takes into account the following requirements:

  • Decoupling with persistence layer: the API should make no assumption about the persistence mechanism used to persist Artifacts. In particular, nothing regarding the parsing/loading should be explicitly exposed thru the API
  • Working copy for Editing in Editor: requires the ability to get a working copy of an Artifact. Optionally, this working copy should be refreshed by changes to that artifact from other clients.
  • Listeners for changes on Artifacts: registering for changes.
  • Resource tracking: ie. if the underlying resource where the artifact is persisted, it should be reloaded and working copies should be notified.
